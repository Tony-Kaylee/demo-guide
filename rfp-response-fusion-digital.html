8. Inventory / Procurement / Order Flow

Assemblies (Single SKU with 4–5 Component SKUs)

Current approach using Packages: Rev.io PSA does not currently support traditional bill-of-materials assemblies with automatic component-level inventory consumption. The Packages feature provides meaningful capability today:

  • Groups multiple products into a named bundle with Bill Profile and Rate Deck Group assignment
  • Display options: Bundled (single line on quote/invoice) or Itemized (each component shown)
  • Provider restrictions configurable per package

Near-term expansion via D-Tools: The D-Tools integration (target June 30, 2026) specifically includes Product/BOM Sync, which brings bill-of-materials data from D-Tools into Rev.io — a direct answer to this need for customers using D-Tools for system design.

Ship-to Information on Vendor PO

Now available. Rev.io PSA added Drop Ship Service Address Auto-Population for Purchase Orders in April 2026. When creating a Drop Ship PO from a ticket, the service address on the ticket is automatically pre-populated as the shipping destination on the PO. Users can override the selection from a dropdown if needed. This directly addresses the pain point you described from Tigerpaw.

Warranty Information in Product Catalog

Available today — at the asset level. When a product is deployed to a customer and registered as a tracked asset, the asset record includes a dedicated Manufacturer Info section with:

  • Manufacturer and Manufacturer Part Number
  • Manufacturer Warranty Expiration date
  • Purchase Provider, Purchase Order Number, Purchase Date
  • Last Service Date / Next Service Date

Warranty tracking is at the individual asset (serialized instance) level rather than the product catalog/SKU level. Additionally, Warranty Expiration Alerts (automated workflow notifications when warranty expiration approaches) is on the roadmap with PR/FAQ complete.

SPA Expiration Tracking & Renewal Management

Available today via platform configuration. While a dedicated SPA management module does not exist, the following approach is used effectively by customers today:

  1. Custom Fields — Add "SPA Expiration Date," "SPA Vendor," and "SPA Type" custom fields to Customer or Asset records via Admin → Custom Fields Manager (custom fields available on Customers, Assets, Tickets, and Purchase Orders)
  2. Workflow Automation — Create a workflow triggered by a date-based condition that fires a configurable number of days before expiration, sending email, SMS, or Teams notifications to the account manager
  3. Recurring Schedules — Create a recurring service ticket for renewal follow-up timed to fire before expiration
  4. System Tags — Tag accounts with active SPAs for quick filtering and bulk visibility across the Customer List

This configuration-based approach is actively used by customers and will continue to be refined as platform capabilities mature.

9. RMA Processing

Current State: A dedicated RMA module is confirmed on the product roadmap. The RMA feature is listed in the Viking 1 release with PR/FAQ specification complete, described as: "Track product returns, replacements/exchanges, and repairs." It is not yet in active development.

Current Process for RMA Handling

Customers manage RMA workflows today using the following platform-native approach:

  1. Create a service ticket to initiate and track the return (ticket type can be labeled for RMA via macros)
  2. Use the Parts & Labor tab to document the returned item and replacement unit
  3. Use Inventory Adjustment to remove the returned unit and add the replacement to stock
  4. Update the Asset record with the replacement serial number and new warranty expiration date

The asset record retains full ticket and notes history from the original unit. Serial number, Manufacturer Warranty Exp, and all purchase fields are manually updatable on the asset record. When the native RMA module ships, it will provide a structured, automated workflow for this process. Your account team can confirm the expected timeline as Viking 1 release planning progresses.

When a product is deployed to a customer and registered as an asset, the asset record includes Manufacturer Warranty Exp, Purchase Date, and all related manufacturer fields in the Manufacturer Info section.

What Is a "Package"?

A Package is a bundled product grouping. It allows you to combine multiple products — recurring services, one-time items, equipment — into a named bundle that can be quoted and sold as a unit. Display can be set to Bundled (single line) or Itemized (each component visible). Packages are used for managed services bundles, connectivity packages, and equipment + service combinations. Package quantity support (entering "number of packages" with automatic quantity multiplication) is in active development.

What Are Macros?

Macros are pre-configured ticket and activity templates that allow technicians to populate a complete ticket in a single click. Two types:

  • New Ticket Macros — pre-fill ticket type, status, priority, description, tech group, and work requested. Examples: Access Control Installation, Emergency On-Site, Network Down, Password Reset, UCaaS Deployment, VPN Access Issue.
  • Activity Macros — pre-fill common note entries, status changes, tag additions, and technician reassignments on existing tickets.

Macros are created and managed in Admin → System Settings → Macros.
