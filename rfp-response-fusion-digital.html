1. Sales Commission Reporting

We want to be transparent rather than oversell this capability.

Current State: Rev.io PSA does not currently include a native, dedicated commission reporting module. The platform does include a Sales Rep field on every customer and opportunity record, and a Profit by Technician dashboard is available in Analytics. However, a structured commission calculation engine with split commissions, gross-profit-based logic, and invoice-date vs. payment-date selection is not available as a built-in feature today.

What Exists Today That Can Be Leveraged

  • Revenue Analysis and Profit by Technician dashboards in Analytics
  • SQL Lab — direct SQL queries against the Rev.io database, which can support custom commission extracts for users comfortable with SQL or for a technical admin to build and save
  • Data export (CSV/Excel/PDF) from any list view for offline commission calculation

8. Inventory / Procurement / Order Flow

Assemblies: Selling a Single SKU with Multiple Component SKUs

Current workaround: Packages. Rev.io PSA does not currently support traditional assemblies (BOM/component-level inventory consumption). However, the Packages feature provides meaningful functionality:

  • A Package groups multiple products (recurring services, one-time items, equipment) into a named bundle
  • Display options: "Bundled" (appears as a single line on quotes/invoices) or "Itemized" (each component shows separately)
  • Bill Profile and Rate Deck Group assignment per package
  • Provider restrictions configurable

Limitation to be aware of: Packages do not automatically consume component-level inventory when the bundle is sold — inventory deduction per component would require manual management. This is an approximation of assembly functionality, not a true BOM with automated component-level inventory consumption.

Ship-to Information on Vendor PO (from Ticket)

We recognize this as a common pain point in Tigerpaw migrations. Whether ship-to information from a service ticket automatically carries over to the vendor PO placed to fulfill it requires confirmation from Rev.io's product team. We recommend requesting a specific answer in writing, and if not yet supported, asking whether it is on the near-term roadmap.

Warranty Information in the Product Catalog

Available today — at the asset level. When a product is deployed to a customer and becomes a tracked asset, the asset record includes a dedicated Manufacturer Info section with:

  • Manufacturer
  • Manufacturer Part Number
  • Manufacturer Warranty Exp (expiration date)
  • Purchase Provider, Purchase Order Number, Purchase Date
  • Last Service Date / Next Service Date

Note: This is tracked at the individual asset (serialized item) level. If you require warranty expiration tracked at the product catalog/SKU level (not instance level), that distinction should be confirmed with Rev.io.

SPA Expiration Tracking & Renewal Management — Workaround

While not available as a dedicated module, the following workaround is practical within the current platform:

  1. Custom Fields — Add "SPA Expiration Date" and "SPA Type" custom fields to Customer or Asset records via Admin → Custom Fields Manager
  2. Workflow Automation — Configure a workflow triggered by a date-based condition that fires X days before expiration, sending email/Teams notification to the account manager
  3. Recurring Schedules — Create a recurring service ticket for renewal follow-up timed to fire before expiration
  4. System Tags — Tag accounts with active SPAs for quick filtering in the Customer List

This workaround is functional but requires discipline to maintain, especially at scale. We recommend discussing SPA volume with Rev.io to determine whether a formal module is planned.

9. RMA Processing

Current State: A dedicated RMA module is not available in Rev.io PSA today. We recognize this is a gap relative to Tigerpaw's current capabilities and want to be direct about it.

Recommended Workaround Today

  1. Create a service ticket (ticket type: Service Call or a configured custom type) to initiate and track the RMA
  2. Use the Parts & Labor tab on the ticket to document the returned item and replacement
  3. Use Inventory Adjustment to remove the returned unit from stock and add the replacement
  4. Use Serial Number tracking to update the asset record with the new unit's serial number
  5. Update Manufacturer Warranty Exp on the asset record to reflect the replacement item's warranty start

On Updating the Asset Record Under RMA

The asset record supports manual updates to Serial Number, Manufacturer Warranty Exp, and all manufacturer/purchase fields. Ticket and notes history from the original asset remains on the record. However, there is no automated one-step "RMA replacement" workflow — this requires manual record updates across inventory and the asset record.

We recommend asking Rev.io for the roadmap ETA on a formal RMA module.

11. Security / Permissions (RBAC)

Role-Based Access Control (RBAC) is natively available in Rev.io PSA today.

What RBAC Means in This Platform

RBAC in Rev.io PSA means that every user is assigned a role, and each role has a granular permission matrix that controls exactly what that user can see and do within the platform. This is not broad module-level on/off — it is fine-grained feature-level control.

How It Works Today

Permission categories15+ categories covering all major modules
Permission granularityIndividual checkbox controls per feature area — not just broad module toggles
AI access controlDedicated "Revii Data Analytics Mode" toggle per role — controls access to AI analytics interface
User groupsTechnicians can be organized into groups (e.g., Tier 1, Tier 2) for ticket routing and dispatch filtering
User capacity100+ user records confirmed in demo environment

Permission categories that can be controlled per role include: customer management, billing, inventory, ticketing, scheduling, projects, admin functions, analytics access, and more.

Distributor / Product Lookup Location

The Commerce Hub module contains the Partners directory. However, as noted in Section 12, live distributor catalog lookup is not currently available. Confirm the roadmap status with Rev.io.

Warranty Information When Adding a Product

Warranty information in Rev.io PSA is tracked at the asset (serialized instance) level, not at the product catalog level. When you add a product to the catalog, there is no warranty field on the catalog record. When that product is deployed to a customer and registered as an asset, the asset record includes Manufacturer Warranty Exp, Purchase Date, and related fields in the Manufacturer Info section.

What Is a "Package"?

A Package is a bundled product grouping in Rev.io PSA. It allows you to combine multiple products — recurring services, one-time items, equipment — into a named bundle that can be quoted and sold as a unit. Packages are configured with a Bill Profile, Rate Deck Group (for metered/VoIP billing), and provider restrictions. Display can be set to "Bundled" (single line on quote/invoice) or "Itemized" (each component shown separately). Packages are commonly used for managed services bundles, connectivity packages, and equipment + service combinations.

What Are Macros and How Are They Used?

Macros are pre-configured ticket templates that allow technicians to populate a complete ticket in a single click, without manually filling in every field. Two types exist:

  • New Ticket Macros — pre-fill ticket type, status, priority, description, tech group, and work requested at ticket creation. Current examples: Access Control Installation, Emergency On-Site, Network Down, New Security System Installation, Password Reset, UCaaS Deployment, VPN Access Issue.
  • Activity Macros — pre-fill common activity entries on existing tickets for frequent actions.

Macros are created and managed in Admin → System Settings → Macros. They are a significant time-saver for high-volume service desks.

Customer Portal Experience

The Customer Portal is available today. It is a self-service web interface accessible to your customers with the following features:

  • View and pay invoices online
  • Update payment methods (credit cards / bank accounts) without calling your team
  • Submit new service tickets directly into your queue
  • View all open tickets with status, severity, and type visible
  • See public notes on active tickets for full transparency

Portal features (Invoices, Tickets, Contacts) are individually toggle-enabled per customer in Admin settings, giving you control over what each customer can access.
